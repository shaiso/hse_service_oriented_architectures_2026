# Маркетплейс — Архитектурное проектирование

## Описание проекта

Архитектура цифрового маркетплейса — платформы, на которой продавцы размещают товары, а покупатели оформляют заказы. Проект включает C4-диаграммы (LikeC4), один сервис в Docker и документацию архитектурных решений.

---

## C4-диаграмма

Диаграммы описаны в формате [LikeC4](https://likec4.dev) и находятся в папке [`architecture/`](./architecture/).

Для просмотра интерактивных диаграмм (из корня репозитория):

```bash
npm install
npm run dev
```

Для экспорта в PNG:

```bash
npm run export
```

## Домены и их ответственность

| Домен | Зона ответственности |
|---|---|
| **User Management** | Регистрация, аутентификация, управление профилями покупателей и продавцов, роли и права доступа |
| **Product Catalog** | Создание, редактирование, удаление товаров; управление категориями; поиск и фильтрация товаров |
| **Order Management** | Оформление заказов, управление жизненным циклом заказа (создан → оплачен → доставляется → завершён), история заказов |
| **Payment** | Расчёт стоимости заказа, инициация платежей через внешний шлюз, учёт транзакций, баланс продавцов |
| **Notification** | Отправка уведомлений покупателям и продавцам о статусах заказов (email, push) |
| **Personalization / Feed** | Формирование персонализированной ленты товаров на основе истории просмотров, покупок и предпочтений пользователя |

---

## Распределение доменов по сервисам

Каждый домен выделен в **отдельный сервис** (1 домен = 1 сервис):

| Сервис | Домен | Обоснование выделения |
|---|---|---|
| **User Service** | User Management | Данные пользователей критичны для безопасности; независимый цикл разработки аутентификации |
| **Catalog Service** | Product Catalog | Самый высоконагруженный на чтение — нуждается в независимом масштабировании |
| **Order Service** | Order Management | Центральный бизнес-процесс с собственным жизненным циклом; требует транзакционной целостности |
| **Payment Service** | Payment | Работа с финансами требует изоляции, отдельного аудита и повышенной безопасности |
| **Notification Service** | Notification | Полностью event-driven; не имеет собственного состояния; легко масштабируется горизонтально |
| **Feed Service** | Personalization | Используется отдельное хранилище (Redis) и алгоритмы рекомендаций; может развиваться независимо |

Дополнительно выделены инфраструктурные компоненты:
- **API Gateway** — единая точка входа, маршрутизация, аутентификация запросов
- **Message Broker (Kafka)** — асинхронная коммуникация между сервисами

**Логика разбиения:** каждый сервис инкапсулирует один бизнес-домен, имеет чёткие API-границы и может быть развёрнут, масштабирован и обновлён независимо от остальных. Разбиение следует принципу Single Responsibility и паттерну Database per Service.

## Границы владения данными

| Сервис | Хранилище | Данные |
|---|---|---|
| **User Service** | PostgreSQL (User DB) | Профили, учётные записи, роли, токены |
| **Catalog Service** | PostgreSQL (Catalog DB) | Товары, категории, описания, цены, остатки |
| **Order Service** | PostgreSQL (Order DB) | Заказы, позиции заказов, статусы, история |
| **Payment Service** | PostgreSQL (Payment DB) | Транзакции, платежи, баланс продавцов |
| **Feed Service** | Redis (Feed Cache) | Персонализированные рекомендации, история просмотров |
| **Notification Service** | — | Stateless; получает данные из событий |

> Каждый сервис владеет своей базой данных. Разделяемых БД между сервисами нет (паттерн Database per Service).

### Взаимодействия сервисов

**Синхронные (REST):**
- API Gateway → User Service (аутентификация)
- API Gateway → Catalog Service (CRUD товаров)
- API Gateway → Order Service (заказы)
- API Gateway → Feed Service (лента)
- Order Service → Catalog Service (проверка наличия товара)
- Order Service → Payment Service (инициация оплаты)
- Payment Service → Payment Gateway (внешний шлюз)
- Notification Service → Email Provider (отправка email)

**Асинхронные (Kafka):**
- Order Service → Message Broker (события заказов: создан, оплачен, отменён)
- Catalog Service → Message Broker (обновления каталога: новый товар, изменение цены)
- Message Broker → Notification Service (уведомления по событиям)
- Message Broker → Feed Service (обновление персонализации)

## Альтернативные варианты декомпозиции

### Вариант A — Модульный монолит

Единое приложение, разделённое на модули по доменам. Каждый модуль (Users, Catalog, Orders, Payments, Notifications, Feed) живёт в одном процессе, но имеет чёткие границы в коде. Общая база данных с разными схемами (schemas) для каждого модуля.

```
[Монолит]
├── module: users      → schema: users
├── module: catalog    → schema: catalog
├── module: orders     → schema: orders
├── module: payments   → schema: payments
├── module: notifications (in-process events)
└── module: feed       → schema: feed
```

### Вариант B — Микросервисы (выбранный)

Каждый домен — отдельный сервис со своей базой данных. Взаимодействие через REST (синхронное) и Kafka (асинхронное). API Gateway как единая точка входа.

```
[API Gateway] → [User Service]    → [User DB]
              → [Catalog Service] → [Catalog DB]
              → [Order Service]   → [Order DB]
              → [Payment Service] → [Payment DB]
              → [Feed Service]    → [Feed Cache]

[Message Broker] ← Order Service, Catalog Service
                 → Notification Service, Feed Service
```

## Trade-off'ы вариантов

| Критерий | Вариант A (Монолит) | Вариант B (Микросервисы) |
|---|---|---|
| **Сложность развёртывания** | Простое: один артефакт, один деплой | Сложное: множество сервисов, оркестрация (Docker Compose / K8s) |
| **Масштабирование** | Только вертикальное или целиком горизонтальное | Независимое масштабирование каждого сервиса |
| **Латентность между модулями** | Минимальная (in-process вызовы) | Сетевые задержки на каждый межсервисный вызов |
| **Транзакционная целостность** | Простые ACID-транзакции в одной БД | Eventual consistency, нужны саги или компенсирующие транзакции |
| **Изоляция отказов** | Сбой в одном модуле может уронить всё приложение | Сбой одного сервиса не влияет на остальные |
| **Независимость команд** | Все работают в одном репозитории, конфликты при мержах | Каждая команда владеет своим сервисом, независимые релизы |
| **Технологическая свобода** | Единый стек для всех модулей | Каждый сервис может использовать свой стек |
| **Операционные затраты** | Низкие: мониторинг одного процесса | Высокие: distributed tracing, service mesh, логирование |

## Обоснование финального выбора

Выбран **Вариант B — Микросервисная архитектура** по следующим причинам:

1. **Разные профили нагрузки.** Catalog Service испытывает пиковые нагрузки на чтение (просмотр товаров), тогда как Payment Service обрабатывает значительно меньше запросов, но каждый критически важен. Масштабировать их независимо — ключевое преимущество.

2. **Изоляция данных и безопасность.** Payment Service работает с финансовыми данными и требует повышенного уровня безопасности и аудита. Физическая изоляция БД снижает поверхность атаки.

3. **Независимость разработки.** Маркетплейс — сложная система с множеством команд. Микросервисы позволяют параллельно развивать домены без конфликтов в кодовой базе.

4. **Event-driven уведомления.** Notification Service по своей природе реактивен — он реагирует на события. Микросервисная архитектура с Message Broker естественно поддерживает этот паттерн.

5. **Эволюция персонализации.** Feed Service может потребовать ML-моделей, другого стека (Python) или специализированных хранилищ. Изоляция в отдельный сервис даёт эту свободу.

> **Компромисс, на который мы идём:** усложнение инфраструктуры (оркестрация, мониторинг, distributed tracing) и eventual consistency вместо ACID. Для маркетплейса с высокой нагрузкой и множеством команд это оправданный trade-off.

## Запуск проекта

### Запуск сервиса

```bash
cd Homework_01

# Запуск user-service в Docker
docker-compose up --build

# Проверка health-check
curl http://localhost:8080/health
# Ожидаемый ответ: {"service":"user-service","status":"ok"}
```

```bash
# Из корня репозитория
npm install

# Интерактивный просмотр
npm run dev
```
