## Спецификация методов

### Создание заказа

**Endpoint:**

```
POST /orders
```

**Тело запроса содержит:**
- `items` — массив позиций, каждая: `{ product_id, quantity }`
- `promo_code` — строка (опционально)

Перед созданием заказа система обязана **последовательно** выполнить следующие проверки. При нарушении любой проверки заказ не создаётся и возвращается соответствующая ошибка.

#### 1. Ограничение частоты создания

- Получить последнюю операцию пользователя с типом `CREATE_ORDER` из таблицы `user_operations`.
- Если с момента её создания прошло менее `N` минут (`N` — конфигурируемый параметр):
    - Возвращается ошибка `ORDER_LIMIT_EXCEEDED`.

#### 2. Проверка активных заказов

- Проверить наличие заказа пользователя в состояниях `CREATED` или `PAYMENT_PENDING`.
- Если такой заказ существует — возвращается ошибка `ORDER_HAS_ACTIVE`.

#### 3. Проверка каталога

- Для каждой позиции проверить, что товар (`product_id`):
    - существует;
    - находится в состоянии `ACTIVE`.
- Если хотя бы один товар не найден или неактивен — возвращается ошибка `PRODUCT_NOT_FOUND` или `PRODUCT_INACTIVE` соответственно.

#### 4. Проверка остатков (stock)

- Для каждой позиции проверить: `product.stock >= quantity`.
- Если хотя бы один товар с недостаточным остатком — возвращается ошибка `INSUFFICIENT_STOCK` с указанием в `details` списка товаров, для которых не хватает остатков (product_id, запрошенное количество, доступный остаток).

#### 5. Резервирование остатков

- При прохождении всех проверок для каждой позиции выполняется: `product.stock -= quantity`.
- Операция резервирования и создание заказа должны выполняться **в одной транзакции** — либо всё, либо ничего.

#### 6. Снапшот цен

- При создании заказа для каждой позиции фиксируется `price_at_order = product.price`.
- Дальнейшие изменения цены товара **не влияют** на уже созданный заказ.

#### 7. Расчёт стоимости

- `total_amount = SUM(price_at_order * quantity)` по всем позициям.
- Если передан `promo_code`:
    - Найти промокод по `code`.
    - Проверить: `active = true`, `current_uses < max_uses`, текущее время находится в интервале `[valid_from, valid_until]`.
    - Если любая проверка не пройдена — возвращается ошибка `PROMO_CODE_INVALID`.
    - Проверить: `total_amount >= min_order_amount`.
    - Если не пройдена — возвращается ошибка `PROMO_CODE_MIN_AMOUNT`.
    - Рассчитать скидку:
        - `PERCENTAGE`: `discount = total_amount * discount_value / 100` (но не более 70% от `total_amount` в ином случае цена 100%)
        - `FIXED_AMOUNT`: `discount = min(discount_value, total_amount)`
    - Итого: `total_amount = total_amount - discount`.
    - Инкрементировать `current_uses` промокода.
- Сохранить `discount_amount` и `total_amount` в заказе.

#### 8. Фиксация операции

- Записать в `user_operations` запись с типом `CREATE_ORDER`.
- Установить `status = CREATED`.

### Обновление заказа

**Endpoint:**

```
PUT /orders/{id}
```

**Тело запроса содержит:**
- `items` — новый массив позиций: `{ product_id, quantity }`

Перед обновлением заказа необходимо выполнить:

#### 1. Проверка владельца

- Заказ должен принадлежать текущему пользователю.
- Иначе — ошибка `ORDER_OWNERSHIP_VIOLATION`.

#### 2. Проверка состояния

- Обновление разрешено **только** в состоянии `CREATED`.
- В остальных состояниях — ошибка `INVALID_STATE_TRANSITION`.

#### 3. Ограничение частоты обновления

- Проверить последнюю операцию пользователя с типом `UPDATE_ORDER`.
- Если прошло менее `N` минут — ошибка `ORDER_LIMIT_EXCEEDED`.

#### 4. Возврат предыдущих остатков

- Для каждой позиции **текущего** заказа: `product.stock += order_item.quantity`.

#### 5. Проверка и резервирование новых позиций

- Для каждой **новой** позиции выполнить те же проверки, что при создании (п.3–5 создания заказа):
    - Товар существует и `ACTIVE`.
    - `product.stock >= quantity`.
- Зарезервировать остатки: `product.stock -= quantity`.

#### 6. Пересчёт стоимости

- Пересчитать `total_amount` по новым позициям.
- Если к заказу был применён промокод — пересчитать скидку (условия промокода проверяются заново, включая `min_order_amount`).
- Если после пересчёта промокод больше не применим (сумма < `min_order_amount`) — скидка снимается, `promo_code_id` обнуляется, `current_uses` промокода декрементируется.

#### 7. Фиксация операции

- Записать в `user_operations` запись с типом `UPDATE_ORDER`.

Всё обновление (возврат старых остатков, резервирование новых, пересчёт, запись) — **в одной транзакции**.

### Отмена заказа

**Endpoint:**

```
POST /orders/{id}/cancel
```

Отдельный эндпоинт (не `PUT`) для явной отмены заказа.

#### 1. Проверка владельца

- Заказ должен принадлежать текущему пользователю.
- Иначе — ошибка `ORDER_OWNERSHIP_VIOLATION`.

#### 2. Проверка состояния

- Отмена разрешена **только** из состояний: `CREATED`, `PAYMENT_PENDING`.
- Иначе — ошибка `INVALID_STATE_TRANSITION`.

#### 3. Возврат остатков

- Для каждой позиции заказа: `product.stock += order_item.quantity`.

#### 4. Возврат использования промокода

- Если к заказу был применён промокод (`promo_code_id IS NOT NULL`):
    - `promo_code.current_uses -= 1`.

#### 5. Установка статуса

- `status = CANCELED`.

Вся операция — **в одной транзакции**.

### 9. Авторизация доступа к API (1 балл)

Реализована JWT-авторизация с двумя типами токенов.

Обязательные эндпоинты:

| Метод | Endpoint | Описание |
|---|---|---|
| `POST` | `/auth/register` | Регистрация пользователя |
| `POST` | `/auth/login` | Аутентификация, возвращает access + refresh token |
| `POST` | `/auth/refresh` | Обновление access token по refresh token |

Требования:
- **Access token** — short-lived (15–30 минут).
- **Refresh token** — long-lived (7–30 дней).
- Все CRUD и бизнес-операции защищены (кроме `/auth/*`).
- При невалидном/просроченном access token — `401` с `error_code: TOKEN_EXPIRED` или `TOKEN_INVALID`.
- При невалидном refresh token — `401` с `error_code: REFRESH_TOKEN_INVALID`.
